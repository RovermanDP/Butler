# CLAUDE.md — Butler 프로젝트 공유 기반

> 이 파일은 **매 Claude Code 세션에 자동 로드**된다.
> 모든 플로우가 공유하는 스택·아키텍처·컨벤션·데이터 모델·범위를 담는다.
> 상세 기능 명세는 `/docs/Butler_PRD.md` 참조. **플로우는 1세션 1플로우로 작업한다.**

## 프로젝트
임대인용 건물관리 앱 **Butler** (PoC). 임대인이 건물·세입자·수납·지출을 한 화면에서 관리하고,
AI 수선비 분담과 자동 알림톡으로 반복 업무·분쟁을 줄인다. 모바일(세로) 기준 UI.

## 기술 스택
- **Frontend:** Vue 3 + Vite. 모바일 세로 화면. 상세 4개 탭은 좌우 스와이프.
- **BaaS:** Supabase — Postgres / Auth / Storage(PDF) / 자동 REST. CRUD·집계는 여기로.
- **Backend:** FastAPI (Python) — AI 수선비 분담, 알림톡 로직(mock)만 담당.
- **AI:** **Anthropic Claude** (`anthropic` Python SDK). 프롬프트(RAG-lite)에 가이드·판례 10~20개 텍스트 주입, 벡터DB 없음.
  - 인증: 환경변수 `ANTHROPIC_API_KEY` (Console 발급, 구독과 별개 사용량 과금).
  - 모델: 기본 `claude-haiku-4-5-20251001`(저렴·빠름), 품질 필요 시 `claude-sonnet-4-6`.
  - 구조화 출력: system에 "JSON만 출력" 지시 → `json.loads` 파싱, 실패 시 고정 시나리오 fallback.

## 책임 분담 (중요)
- **CRUD·집계 → Supabase 직결** (Vue → Supabase JS client). 집계 지표는 View/RPC.
- **AI 분담·알림톡 → FastAPI 경유** (Vue → FastAPI → LLM/외부).
- **세입자 등록 → 대부분 CRUD(Supabase) + 결정적 계산만 별도.** 종료일 자동 +1년·첫 납부일(선불/후불)·회차 자동 생성은 **LLM이 아니라** 프론트 composable 또는 Supabase RPC로 처리. 등록 결과(세입자·계약·회차·증빙)는 Supabase에 저장.
- 새 백엔드 엔드포인트는 단순 CRUD면 만들지 말고 Supabase를 쓴다.

## 데이터 모델 (요약 — 전체 DDL은 PRD 3장)
`buildings` / `tenants`(개인·사업자 구분) / `contractors`(공동계약자, PoC 선택) /
`contracts`(세입자당 다중 계약·월세/전세) / `payments`(회차별) /
`expenses`(is_repair=true면 AI분담) / `repair_allocations` / `notifications`(발송 mock).
- **세입자 등록 신규 필드(상세는 PRD 3장):** `tenants.tenant_type`('개인'|'사업자')·사업자명/담당자명, `contracts`에 계약형태(월세/전세)·보증금(전세 필수)·관리비·기타비용1/2 + 항목별 부가세 플래그·납부일·선불/후불·첫 납부일·입금자명·**증빙 설정 5종**(+조건부 필드)·계약서/사업자등록증 URL(Storage).
- **증빙(proof) 두 종류 혼동 금지:** ① `expenses.proof_type`(지출 증빙: 세금계산서/현금영수증(개인)/간이영수증) ② 세입자 등록 시 **임대료 증빙 발급 설정**(`contracts.proof_kind`: 해당없음/현금영수증(개인소득공제용)/현금영수증(사업자증빙용)/세금계산서/계산서). 서로 다른 개념.
- 집계(입주율·미납·임대수익·매월 입금 예정 총액 등)는 컬럼이 아니라 **View/RPC로 계산**한다.
- 모든 화면 수치는 와이어프레임 기준 **시드 데이터**를 쓴다.

## 컨벤션
- 테이블·컬럼: snake_case. 금액: bigint(원 단위). 날짜: date / timestamptz.
- 상태값 한글 문자열 그대로 사용: 계약 status '계약중'|'만료'|'종료', 수납 status '대기'|'미납'|'완납'.
- 세입자 등록 enum(한글 그대로): tenant_type '개인'|'사업자', 계약형태(lease_type) '월세'|'전세', 납부 조건(payment_timing) '선불'|'후불', 과거 미납 '미납없음'|'1개월'|'2개월', 증빙 설정(proof_kind) '해당없음'|'현금영수증(개인소득공제용)'|'현금영수증(사업자증빙용)'|'세금계산서'|'계산서'.
- 통화 표시: `1,850,000 원` 형식. 비율: 정수 % (예: 70).
- 컴포넌트: 화면 단위 + 재사용 카드/태그 컴포넌트 분리.

## 디자인 토큰 (Butler Design System — override 기준 = 실제 렌더링)
> 주색은 단일 Slate Blue. White & Light Gray 베이스, 얇은 선, radius 10px 중심.
> ⚠ 파일 위쪽 민트(#3ECCA0) :root는 폐기된 구버전. 304줄 override가 실제 적용값.
- **폰트:** Pretendard. 화면 배경 흰색(#fff), 본문 텍스트 --ink #1F1F24.
- **팔레트:**
  - accent(주색·단일 포인트) #3A6EA5 / deep #2A5689 / soft #E3ECF6
  - ok(완납·정상) #1A9C66 / soft #DFF3E8
  - danger(에러·미납) #D63B3B / soft #FBE0E0
  - warn(즐겨찾기·증빙) #C2870A / soft #FBECD4
  - gray 1~7: #F4F4F7 #E6E6EC #D4D4DC #A8A8B3 #777783 #5E5E68 #3F3F48
  - ink 변형(본문 보조): ink-soft #5E5E68(=gray-6) / ink-mute #A8A8B3(=gray-4)
- **radius:** 10px를 중심 기조로. (카드·입력·버튼 모두 10 안팎으로 통일)
- **선:** 얇은 1px, --line #E6E6EC.
- **그림자:** 매우 약하게. 0 4px 16px rgba(31,31,36,.04), 0 1px 2px rgba(31,31,36,.04)
- **상태 색 매핑:** 정상·완납=ok(green) / 미납·에러=danger(red) / 즐겨찾기·증빙=warn(amber) / 모든 강조·버튼=accent(slate blue 단일).
- ⚠ 보드 CSS 금지: 어두운 보드 배경·폰 목업·화살표 커넥터·주석은 발표용 장식. 앱에 옮기지 말 것.
- **세입자 등록 신규 컴포넌트(같은 토큰 사용):** 스텝 도트(.stepdots), 개인/사업자·월세/전세 세그먼트 토글(.seg), 계약기간 캘린더 모달(.datemodal)·종료일 자동 +1년 뱃지(.auto-1y), 휠 피커(.wheel, 입금예정일/과거미납), 하단 고정 액션바(.formfoot 다음/뒤로), 입금자명 칩(.namebox)·첫 납부일 박스(.firstpay), 증빙 모달 리스트(.sheet-list), 완료 화면(.done-*). 모두 단일 accent(slate blue) 기조 유지.


- **기존 패턴 우선:** 새 코드 전에 유사 기존 구현(폴더 구조·네이밍·레이어 분리)을 먼저 보고 그대로 따른다. 동작만 하고 구조 안 맞는 코드 금지.
- **단일 진실 공급원(OSoT):** 타입·상수·설정·비즈니스 로직은 한 곳에만 정의. 새로 짜기 전 동일/유사 정의가 있는지 확인하고 있으면 참조.
- **에러 처리:** happy path만 짜지 말 것. 예외·엣지(연속 클릭·잘못된 입력·네트워크 실패) 처리, 의미 있는 사용자 피드백 제공. 에러를 로그로 삼키거나 임시방편으로 숨기지 말 것. TS는 `any` 금지, Python은 bare `except`·무음 처리 금지.
- **단일 책임(SRP):** 함수/파일은 한 역할만. UI·비즈니스·데이터 로직 혼합 금지. 이름만 봐도 역할이 보이게.
- **공유 분리:** 재사용 로직은 Vue `composables/`·`utils/`, FastAPI `services/`·`utils/`로. 새로 짜기 전 기존 공유 유틸 먼저 확인.
- ⚠ **PoC 단서:** 위 원칙은 *명백히 중복/혼재할 때* 적용. 선제적 추상화·과도한 파일 분할은 지양하고 데모 일정을 우선한다.

## 범위 경계 (반드시 준수)
- ❌ 실결제 / PG / 자동 수납 연동 없음. (완료 화면 [카드로 결제하기]는 UI only)
- ❌ **카카오 알림톡 실제 발송 없음 → mock 처리** (status='mock_sent'). 템플릿 심사 기간 때문.
     히스토리·미리보기·발송 스케줄 UI까지만 진짜 구현.
- ✅ **세입자 등록은 범위 内 (멀티스텝 위저드 · 직접 등록 경로).** 단:
  - ❌ '임대차 계약서 추천'(계약서 업로드 → 자동 추출/OCR) 경로는 **버튼·안내 카드 teaser만**, 자동완성 미구현.
  - ❌ 계약서·사업자등록증 업로드는 **Storage 저장까지만**(진위 검증·OCR·국세청 조회 없음).
  - ❌ 상세 5번째 '공지' 탭(세입자 연동/공지 발송)은 **teaser**, PoC 범위 외.
  - 종료일 +1년·첫 납부일·회차 생성은 결정적 계산(LLM 아님).
- ❌ 세입자용 앱 없음. ❌ 학습형 AI 자체 학습 없음. ❌ 다지역·회계 연동 없음.
- 인증은 단일 임대인 계정 수준(Supabase Auth).

## AI 수선비 분담 — 데모 안정성
- 실시간 LLM 호출 + **고정 시나리오 병행**. LLM 실패 시 고정값 fallback.
- 대표 고정 시나리오: 냉장고 / 노후·자연마모 / 7년 → **임대인 70 : 임차인 30**.
- 결과에 **LH 가이드라인·대법원 판례 근거 출처 + 면책 문구** 항상 노출.
- ⚠ 백엔드 실행 셸에 `ANTHROPIC_API_KEY`가 있으면 그 셸의 Claude Code가 API 과금될 수 있음 — 빌드용/앱 실행용 키 환경 분리.

## 알림톡 발송 스케줄 (UI 표기 기준)
- D-1·D-day 납부 안내(18:00) / D+2·4·7 미납 안내(13:00) /
  D-60 연장 여부(AM 10:00) / D-3 계약 종료(AM 10:00).
- 종류별 색상 태그: 납부 / 미납 / 연장 / 종료.

## 작업 원칙
- 한 번에 한 플로우만. "PRD 4.x만 구현, 다른 플로우는 건드리지 말 것." (세입자 등록은 4.5 = Flow E)
- 플로우 완료 시 PRD의 해당 체크리스트로 검증 → 커밋 → 새 세션.
- **기존 컴포넌트 재사용:** Flow A의 '건물 관리' 바텀시트는 화면 9(세입자 등록 활성)에서 그대로 재사용한다 — 복제 금지, 세입자 등록 옵션 활성 여부를 prop/상태로 분기.
- 공유 변경(테이블·컨벤션)은 이 파일을 먼저 갱신.
