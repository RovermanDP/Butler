# CLAUDE.md — Butler 프로젝트 공유 기반

> 이 파일은 **매 Claude Code 세션에 자동 로드**된다.
> 모든 플로우가 공유하는 스택·아키텍처·컨벤션·데이터 모델·범위를 담는다.
> 상세 기능 명세는 `/docs/Butler_PRD.md` 참조. **플로우는 1세션 1플로우로 작업한다.**

## 프로젝트
임대인용 건물관리 앱 **Butler** (PoC). 임대인이 건물·세입자·수납·지출을 한 화면에서 관리하고,
AI 수선비 분담과 자동 알림톡으로 반복 업무·분쟁을 줄인다. 모바일(세로) 기준 UI.

## 기술 스택
- **Frontend:** Vue 3 + Vite. 모바일 세로 화면. 상세 4개 탭은 좌우 스와이프.
- **BaaS:** Supabase — Postgres / Auth / Storage(PDF) / 자동 REST. CRUD·집계는 여기로.
- **Backend:** FastAPI (Python) — AI 수선비 분담, 알림톡 로직(mock)만 담당.
- **AI:** **Anthropic Claude** (`anthropic` Python SDK). 프롬프트(RAG-lite)에 가이드·판례 10~20개 텍스트 주입, 벡터DB 없음.
  - 인증: 환경변수 `ANTHROPIC_API_KEY` (Console 발급, 구독과 별개 사용량 과금).
  - 모델: 기본 `claude-haiku-4-5-20251001`(저렴·빠름), 품질 필요 시 `claude-sonnet-4-6`.
  - 구조화 출력: system에 "JSON만 출력" 지시 → `json.loads` 파싱, 실패 시 고정 시나리오 fallback.

## 책임 분담 (중요)
- **CRUD·집계 → Supabase 직결** (Vue → Supabase JS client). 집계 지표는 View/RPC.
- **AI 분담·알림톡 → FastAPI 경유** (Vue → FastAPI → LLM/외부).
- 새 백엔드 엔드포인트는 단순 CRUD면 만들지 말고 Supabase를 쓴다.

## 데이터 모델 (요약 — 전체 DDL은 PRD 3장)
`buildings` / `tenants` / `contracts`(세입자당 다중 계약) / `payments`(회차별) /
`expenses`(is_repair=true면 AI분담) / `repair_allocations` / `notifications`(발송 mock).
집계(입주율·미납·임대수익 등)는 컬럼이 아니라 **View/RPC로 계산**한다.
모든 화면 수치는 와이어프레임 기준 **시드 데이터**를 쓴다.

## 컨벤션
- 테이블·컬럼: snake_case. 금액: bigint(원 단위). 날짜: date / timestamptz.
- 상태값 한글 문자열 그대로 사용: 계약 status '계약중'|'만료'|'종료', 수납 status '대기'|'미납'|'완납'.
- 통화 표시: `1,850,000 원` 형식. 비율: 정수 % (예: 70).
- 컴포넌트: 화면 단위 + 재사용 카드/태그 컴포넌트 분리.

## 디자인 토큰 (와이어프레임 기준 — 똑같이 재현)
> 화면별 레이아웃·구성요소는 PRD 4·5장 + `docs/Butler_Wireframe.html`(해당 화면만) 참조.
- **폰트:** Pretendard. 앱 화면 배경 흰색(#fff), 본문 텍스트 `--ink #1E2630`.
- **팔레트:**
  - mint(주색) `#3ECCA0` / deep `#2BB489` / soft `#E6F8F1`
  - blue(스와이프·강조) `#4A7DFF` / soft `#E8EEFF`
  - red(에러·미납) `#F2547D` / soft `#FDE7EE` · rose `#CD7B74`
  - amber(즐겨찾기·증빙) `#C8A23B` / soft `#FBF1DA`
  - gray 1~7: `#F2F4F6 #E6E9ED #C7CDD4 #9AA4AF #6B7682 #4F5A66 #3A4452`
- **radius:** 카드 18 / 바텀시트 24(상단) / 입력 13 / 버튼 14 / 프로모션 16 / 태그·뱃지 8 / FAB 원형.
- **상태 색 매핑:** 완납·정상=mint, 미납·에러=red, 스와이프 탭 활성=blue, 즐겨찾기·증빙=amber.
- ⚠ **보드 CSS 금지:** 와이어프레임의 어두운 보드 배경·폰 목업 프레임·화살표 커넥터·주석은 *발표용 장식*이다. 앱에 옮기지 말 것. 가져올 것은 폰 화면(`.phone .scr`) 내부 구성과 위 `:root` 토큰뿐.


- **기존 패턴 우선:** 새 코드 전에 유사 기존 구현(폴더 구조·네이밍·레이어 분리)을 먼저 보고 그대로 따른다. 동작만 하고 구조 안 맞는 코드 금지.
- **단일 진실 공급원(OSoT):** 타입·상수·설정·비즈니스 로직은 한 곳에만 정의. 새로 짜기 전 동일/유사 정의가 있는지 확인하고 있으면 참조.
- **에러 처리:** happy path만 짜지 말 것. 예외·엣지(연속 클릭·잘못된 입력·네트워크 실패) 처리, 의미 있는 사용자 피드백 제공. 에러를 로그로 삼키거나 임시방편으로 숨기지 말 것. TS는 `any` 금지, Python은 bare `except`·무음 처리 금지.
- **단일 책임(SRP):** 함수/파일은 한 역할만. UI·비즈니스·데이터 로직 혼합 금지. 이름만 봐도 역할이 보이게.
- **공유 분리:** 재사용 로직은 Vue `composables/`·`utils/`, FastAPI `services/`·`utils/`로. 새로 짜기 전 기존 공유 유틸 먼저 확인.
- ⚠ **PoC 단서:** 위 원칙은 *명백히 중복/혼재할 때* 적용. 선제적 추상화·과도한 파일 분할은 지양하고 데모 일정을 우선한다.

## 범위 경계 (반드시 준수)
- ❌ 실결제 / PG / 자동 수납 연동 없음.
- ❌ **카카오 알림톡 실제 발송 없음 → mock 처리** (status='mock_sent'). 템플릿 심사 기간 때문.
     히스토리·미리보기·발송 스케줄 UI까지만 진짜 구현.
- ❌ 세입자용 앱 없음. ❌ 학습형 AI 자체 학습 없음. ❌ 다지역·회계 연동 없음.
- 인증은 단일 임대인 계정 수준(Supabase Auth).

## AI 수선비 분담 — 데모 안정성
- 실시간 LLM 호출 + **고정 시나리오 병행**. LLM 실패 시 고정값 fallback.
- 대표 고정 시나리오: 냉장고 / 노후·자연마모 / 7년 → **임대인 70 : 임차인 30**.
- 결과에 **LH 가이드라인·대법원 판례 근거 출처 + 면책 문구** 항상 노출.
- ⚠ 백엔드 실행 셸에 `ANTHROPIC_API_KEY`가 있으면 그 셸의 Claude Code가 API 과금될 수 있음 — 빌드용/앱 실행용 키 환경 분리.

## 알림톡 발송 스케줄 (UI 표기 기준)
- D-1·D-day 납부 안내(18:00) / D+2·4·7 미납 안내(13:00) /
  D-60 연장 여부(AM 10:00) / D-3 계약 종료(AM 10:00).
- 종류별 색상 태그: 납부 / 미납 / 연장 / 종료.

## 작업 원칙
- 한 번에 한 플로우만. "PRD 4.x만 구현, 다른 플로우는 건드리지 말 것."
- 플로우 완료 시 PRD의 해당 체크리스트로 검증 → 커밋 → 새 세션.
- 공유 변경(테이블·컨벤션)은 이 파일을 먼저 갱신.
